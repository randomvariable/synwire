# Research: Generic State Trait

**Branch**: `002-generic-state-trait` | **Date**: 2026-03-15

## R1: State Trait Supertraits — Clone vs Reference Semantics

**Decision**: `State: Send + Sync + Clone + Serialize + DeserializeOwned + 'static`

**Rationale**: The Pregel superstep loop passes state through sequential nodes. Each node receives ownership of the state and returns an updated version. `Clone` is needed for two cases: (1) checkpointing (clone before passing to next node so checkpoint gets a snapshot), (2) conditional edges (condition receives `&S`, node receives `S`). Since the current Value-based model already clones Value at the same points, Clone adds no new overhead.

**Alternatives considered**:
- `State` without `Clone`, using `Arc<S>` for shared access — rejected because nodes need mutable access to state, and `Arc<Mutex<S>>` adds unnecessary complexity and contention.
- `State` with `Copy` — rejected because realistic state types (containing Vec, String, HashMap) cannot implement Copy.

## R2: from_channels Design — Generated vs Manual

**Decision**: `from_channels` is generated by `#[derive(State)]`. Manual implementation is possible for advanced cases by implementing the `State` trait directly.

**Rationale**: The `#[derive(State)]` macro has access to field names and types at compile time. It can generate `from_channels` by deserialising each channel's `serde_json::Value` into the field's declared type via `serde_json::from_value::<FieldType>(channel.get())`. This is the same pattern serde's own derive macros use.

**Alternatives considered**:
- Require manual `from_channels` — rejected because it's boilerplate that the macro can generate correctly.
- Use a `ChannelMap` abstraction instead of raw `HashMap<String, Box<dyn BaseChannel>>` — deferred. The current BaseChannel trait returns `Option<&Value>` from `get()`, which is sufficient for deserialisation. A typed channel abstraction can be added later without breaking changes.

## R3: channels() Return Type — HashMap vs Vec

**Decision**: `fn channels() -> Vec<(String, Box<dyn BaseChannel>)>` (matching current derive output)

**Rationale**: The current `#[derive(State)]` macro already returns `Vec<(String, Box<dyn BaseChannel>)>`. Changing to `HashMap` would require an extra allocation and provides no benefit since channels are iterated, not randomly accessed, during graph construction. The spec's `HashMap` signature was aspirational — the Vec form is more efficient and already implemented.

**Alternatives considered**:
- `HashMap<String, Box<dyn BaseChannel>>` as spec says — rejected for the reason above. Easy to convert if needed downstream.

## R4: Prebuilt Node Generics Strategy

**Decision**: Prebuilt nodes that need access to typed state fields use trait bounds. Nodes that only need JSON-level access (HttpRequestNode, TemplateTransformNode) use a `StateAccessor<S>` pattern to extract/inject the relevant field via closures.

**Rationale**: Some prebuilt nodes (IfElseNode, LoopNode) need to inspect state to make routing decisions — these benefit from typed state. Others (HttpRequestNode, TemplateTransformNode) operate on string templates and JSON payloads — they don't benefit from knowing the state type. For the latter, a `StateAccessor` closure (`Fn(&S) -> Value` / `Fn(&mut S, Value)`) bridges between typed state and the node's internal Value logic, avoiding the need to rewrite their internals.

**Alternatives considered**:
- Make all nodes fully generic with no Value fallback — rejected because it would require rewriting HttpRequestNode to understand arbitrary state types, which isn't meaningful (it sends HTTP requests with string URLs).
- Keep prebuilt nodes as Value-only and wrap them — considered viable but adds wrapping boilerplate at every usage site.

## R5: ToolNode Generic Strategy

**Decision**: `ToolNode` remains Value-based internally (it works with JSON tool calls and results). It gains a generic `into_node_fn<S: State>()` method that reads `messages` from `S` via serde, and writes tool results back by serialising into the state.

**Rationale**: Tool calls and results are JSON by nature (they come from LLM responses as JSON objects). The ToolNode's internal logic of parsing tool_calls, invoking tools, and constructing tool-response messages is inherently Value-based. The generic boundary is at the edges: extracting messages from `S` and injecting tool messages back into `S`.

**Alternatives considered**:
- Require state types to implement a `HasMessages` trait — considered for M2 (the `MessagesState` pattern) but too constraining for the generic `ToolNode` which should work with any state that has a `messages` field.

## R6: MessagesState Location

**Decision**: `MessagesState` is defined in `synwire-orchestrator::messages` module, not in `synwire-core`.

**Rationale**: `MessagesState` depends on the `State` trait which is defined in `synwire-orchestrator`. It uses `synwire_core::messages::Message` for the field type. Defining it in core would create a circular dependency (core -> orchestrator for State trait). The orchestrator crate is the natural home since it owns the graph execution model.

**Alternatives considered**:
- Define in `synwire-core` with State trait also in core — rejected because channels are in orchestrator and State depends on `BaseChannel`.
- Define in a new `synwire-state` crate — rejected as unnecessary indirection for one type.

## R7: Error Type Extension

**Decision**: Add `GraphError::DeserializationError { field: String, message: String }` variant.

**Rationale**: `from_channels` needs a specific error when a channel value can't be deserialised into the target field type. The existing `InvalidUpdate` variant is too generic — it doesn't carry the field name. A specific variant enables better error messages: "failed to deserialise field 'messages': expected array, got string".

**Alternatives considered**:
- Reuse `InvalidUpdate` with a descriptive message — rejected because callers can't programmatically distinguish deserialisation failures from other invalid updates.

## R8: Backward Compatibility for Value-Based Graphs

**Decision**: Provide a `ValueState` type that wraps `serde_json::Value` and implements `State`. This lets existing Value-based code continue working with `StateGraph<ValueState>` and minimal changes (wrap input in `ValueState`, unwrap output).

**Rationale**: The existing examples, tests, and ReAct agent tests all use raw Value. A clean migration path avoids breaking all test code at once. `ValueState` is a thin wrapper — `struct ValueState(pub serde_json::Value)` with `State` impl that returns a single passthrough channel.

**Alternatives considered**:
- Implement `State` for `serde_json::Value` directly — rejected because `Value` is an external type (orphan rule) and its `channels()` semantics are ambiguous (what channels does an arbitrary JSON object have?).
- Rewrite all tests to use typed state immediately — considered viable but increases the change scope significantly. ValueState provides a migration bridge.
