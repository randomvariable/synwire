# Research: GoReleaser MCP Server Binaries & Book Publishing Fix

## 1. GoReleaser Prebuilt Builder

**Decision**: Use GoReleaser v2 `builder: prebuilt` with binaries staged in `artifacts/<os>_<arch>[_variant]/`.

**Rationale**: Rust build tooling (Cargo) is separate from GoReleaser. The cleanest integration is to build binaries in a GitHub Actions matrix, upload them as workflow artifacts, then hand off to GoReleaser in "release only" mode. GoReleaser handles archive packaging, checksum generation, changelog, and GitHub Release creation.

**Key constraint**: GoReleaser deletes `dist/` before running. Pre-built binaries must be staged in a different directory (`artifacts/` by convention). The prebuilt path template must use `{{ with .Amd64 }}_{{ . }}{{ end }}` to produce the `_v1` suffix that GoReleaser expects for amd64 targets.

**Prebuilt path template**: `artifacts/{{ .Os }}_{{ .Arch }}{{ with .Amd64 }}_{{ . }}{{ end }}/synwire-mcp-server{{ .Ext }}`

**Alternatives considered**: cargo-dist (purpose-built for Rust but spec requires GoReleaser), inline `cargo build` hooks in GoReleaser (doesn't support cross-compilation to macOS from Linux).

## 2. Linux aarch64 Cross-Compilation

**Decision**: `cargo-zigbuild` on `ubuntu-latest` for both Linux targets (`x86_64-unknown-linux-musl` and `aarch64-unknown-linux-musl`).

**Rationale**:
- No Docker daemon required (no Docker-in-Docker complexity)
- Zig ships its own cross-linker; no sysroot configuration needed
- musl targets produce fully statically linked binaries (satisfies FR-009)
- Works for both x86_64 (native) and aarch64 (cross) on the same runner

**Alternative considered — `cross`**: Requires Docker-in-Docker on GitHub Actions runners, which is flaky and slow. Rejected.

**Alternative considered — `gcc-aarch64-linux-gnu` + musl sysroot**: Requires manual sysroot configuration and breaks for crates with C dependencies. Rejected.

**Musl requirement**: Both Linux targets use `-musl` variants to satisfy FR-009 (static linking). The project uses `rustls` (not OpenSSL), so no vendored OpenSSL is needed.

## 3. macOS Ad-Hoc Signing

**Decision**: `codesign --sign - --options runtime --timestamp` on macOS runners after cargo build.

**Rationale**: `--options runtime` enables the hardened runtime flag, which is required for Sequoia 15.1+ compatibility. `--timestamp` embeds Apple's secure timestamp. This is more robust than bare `codesign -s -`.

**Important limitation**: Ad-hoc signing does not remove the quarantine extended attribute that macOS sets on files downloaded by a browser. Users who download directly from GitHub Releases on Sequoia 15.1+ may still see a Gatekeeper prompt and must run `xattr -d com.apple.quarantine /path/to/binary`. This must be documented in release notes and installation instructions.

**Install path recommendation**: Users installing via `curl | sh` pipelines do not encounter quarantine (quarantine is set by the browser, not curl). A Homebrew tap is the recommended install path for macOS users; Homebrew re-signs binaries automatically.

**Full notarization**: Requires a paid Apple Developer ID certificate + `xcrun notarytool`. Not required for this feature but the `--options runtime` flag preserves the path to add notarization later.

**Alternatives considered**: Full notarization (requires Apple Developer account — out of scope), no signing (would block execution on Sequoia 15.1+ entirely — rejected per FR-011).

## 4. Release Workflow Architecture

**Decision**: Three-phase workflow in `release.yml`:
1. Matrix build jobs (4 runners: ubuntu × 2, macos × 2) — compile, sign, upload artifacts
2. GoReleaser job (depends on all matrix jobs) — download artifacts, create release

**Why separate runners for macOS**: macOS cross-compilation from Linux requires the macOS SDK and toolchain. The only practical approach without osxcross is native macOS runners. For the two Linux targets, a single ubuntu-latest runner with cargo-zigbuild suffices.

**Artifact staging**: Each matrix job uploads to a named artifact `synwire-mcp-server-<os>-<arch>`. The release job uses `download-artifact@v4` with `merge-multiple: true` to flatten all artifacts into a single `artifacts/` tree matching the GoReleaser prebuilt path template.

**`fetch-depth: 0`**: Required on the release job checkout so GoReleaser can read git history for changelog generation.

## 5. Release Body (Crate Table)

**Decision**: Use GoReleaser's `release.footer` template to include the crate table currently generated by the `github-release` job.

**Rationale**: GoReleaser generates the changelog section; the crate table is appended via `footer`. This preserves the existing release body content while removing the manual `softprops/action-gh-release` step.

## 6. Pages Fix

**Decision**: Change `path: docs/book` to `path: docs/book/html` in `pages.yml` line 38.

**Root cause confirmed**: mdBook writes HTML output to `<build-dir>/html/` (not `<build-dir>/` directly). The workflow uploads `docs/book/` which contains `html/` and `linkcheck/` subdirectories with no root `index.html`. GitHub Pages returns 404 because there is no `index.html` at the artifact root.

**Verification**: Local `mdbook build docs` confirms output at `docs/book/html/index.html`. The last successful CI run logs show `./html/404.html`, `./html/reference/...` etc. at the root of the uploaded archive.

---

## 7. Changelog Configuration

**Decision**: `use: github` with conventional commit groups (Breaking Changes, Features, Bug Fixes, Performance, Documentation, Other). Exclude `chore(deps):`, `ci:`, merge commits.

**Rationale**: `use: github` provides richer commit metadata and contributor attribution. Conventional commit grouping matches the project's commit standards from the constitution.
