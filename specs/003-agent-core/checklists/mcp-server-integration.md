## Scenario Coverage

- [x] CHK024 - Are requirements defined for the MCP server startup sequence (what happens before the first tool call — initialization, index loading, skill discovery)? [Coverage, Gap]
  > ✅ FR-888m: 5-step sequence (parse CLI → init StorageLayout → connect/start daemon → load skills → register tools → accept requests). 10s budget.
- [x] CHK025 - Are requirements defined for what tools are available before `--project` is set or `index` is called? (e.g., only `clone_repo` and `index`?) [Coverage, Spec §Edge Case]
  > ✅ FR-888k: minimal set (index, clone_repo, VFS file ops). Search tools return "call index first" error.
- [x] CHK026 - Are requirements defined for tool call timeout behaviour — what happens if a tool call (e.g., `grep` on a huge repo) takes too long? [Coverage, Gap]
  > ✅ FR-888l: 120s configurable timeout, error result to client, daemon-side ops not cancelled.
- [x] CHK027 - Are requirements defined for how the MCP server reports its capabilities to the client (tool list, server info, protocol version)? [Coverage, Gap]
  > ✅ FR-888o: `initialize` response with tools list + serverInfo (name, version). FR-888a: protocol version.
- [x] CHK028 - Are requirements for `--lsp` and `--dap` integration defined for the case where the language server/debug adapter crashes mid-session? [Coverage, Spec §FR-865–866]
  > ✅ FR-888p: auto-restart up to 3 times with exponential backoff. Tools return errors during restart.

## Edge Case Coverage

- [x] CHK029 - Is the behaviour defined when `--project` points to a path that doesn't exist? [Edge Case, Gap]
  > ✅ FR-888q: "fail at startup with a clear error"
- [x] CHK030 - Is the behaviour defined when the skills directory contains malformed `SKILL.md` files — does the server start with partial skills or fail entirely? [Edge Case, Gap]
  > ✅ FR-888r: "skipped with warning log per file. Server starts with all valid skills. One bad skill does not block others."
- [x] CHK031 - Is the behaviour defined when disk space runs out during indexing? [Edge Case, Gap]
  > ✅ Edge case added: daemon detects write failures, marks index incomplete, next attempt resumes via hash registry.
- [x] CHK032 - Is the behaviour defined when two MCP server instances try to index the same project simultaneously? [Edge Case, Spec §Edge Case]
  > ✅ FR-825a: single daemon handles all indexing. Two MCP servers route to the same daemon. The daemon serialises indexing per WorktreeId internally.
- [x] CHK033 - Is the behaviour defined when a tool call arrives while the previous tool call is still in progress (concurrent tool calls from the same client)? [Edge Case, Gap]
  > ✅ FR-888s: "Concurrent tool calls supported — forwarded to daemon in parallel. Results returned as they complete."
- [x] CHK034 - Is the behaviour defined when `StorageLayout` directories have incorrect permissions (not writable)? [Edge Case, Gap]
  > ✅ FR-888t: "fail at startup with a clear error identifying which directory is not writable"

## Non-Functional Requirements

- [x] CHK035 - Are memory usage requirements specified for the MCP server process at rest (no active operations)? [Gap]
  > ✅ MCP server is a thin proxy (FR-825d) — heavy state is in the daemon. MCP server at rest is effectively just a UDS connection + skill metadata. Daemon memory covered by SC-144 (<2GB). Acceptable: MCP server memory is negligible by design.
- [x] CHK036 - Are startup time requirements specified (time from process spawn to first tool being available)? [Gap]
  > ✅ FR-888m: "Startup MUST complete within 10 seconds excluding daemon start time"
- [x] CHK037 - Are logging requirements specific enough to diagnose "why is semantic search returning bad results?" as stated in SC-143? [Completeness, Spec §SC-143]
  > ✅ FR-878: "Log entries for search operations MUST include: query, result count, and latency". SC-143 confirmed.
- [x] CHK038 - Is the `RUST_LOG` environment variable interaction with `--log-level` specified (precedence)? [Gap, Spec §FR-875]
  > ✅ FR-888n: "RUST_LOG takes precedence over --log-level. --log-level sets the default; RUST_LOG overrides with per-module granularity."
- [x] CHK039 - Are requirements for binary size specified or bounded (affects download/install experience)? [Gap]
  > ✅ Not specified as a hard requirement. The binary includes fastembed (33MB model downloaded at runtime), tree-sitter grammars, and clap. Size is an implementation concern, not a spec requirement. Acceptable to leave unspecified — the model download dominates anyway.
